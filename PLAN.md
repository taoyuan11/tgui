# tgui 项目计划书

## 1. 项目目标

`tgui` 的目标是构建一个 Rust 原生 GUI 库，采用声明式 DSL（Domain Specific Language）与 MVVM（Model-View-ViewModel）架构模式，基于 `winit + wgpu` 提供现代桌面界面能力。

本项目首版聚焦“轻量运行时 + 可扩展架构”路线，优先完成从应用启动、窗口管理、状态绑定到基础渲染的最小闭环，而不是一开始就追求“大而全”的 GUI 框架能力。

项目定位如下：

- Rust 原生：优先遵循 Rust 的所有权、类型安全和零成本抽象思路。
- 声明式：UI 以声明式组件树描述，而不是命令式逐步拼装。
- MVVM 驱动：状态、命令与视图更新围绕 `ViewModel` 建立。
- 桌面三平台预留：目标覆盖 Windows、macOS、Linux，首版先完成统一架构与主要抽象。

## 2. 设计原则

### 2.1 轻量优先

首版不追求覆盖所有 GUI 能力，而是优先保证主链路完整：应用启动、事件输入、状态变更、布局计算、渲染输出能够稳定协作。

### 2.2 可测试

尽量将运行时、绑定系统、布局逻辑与平台细节解耦，使核心逻辑具备单元测试和集成测试空间。尤其是 `ViewModel`、绑定、脏标记、布局计算不应依赖窗口后端才能验证。

### 2.3 渲染与状态解耦

渲染器只负责消费 UI 树生成的绘制数据和提交 GPU 命令，不直接承担业务状态管理职责。状态更新应由 `ViewModel` 与绑定层驱动，避免渲染层反向污染 UI 逻辑。

### 2.4 最小可用优先

优先交付一个可以运行示例应用、支持基础控件和响应式更新的 v1，而不是提前引入复杂动画、可视化设计器、资源编辑器等高成本能力。

### 2.5 平台抽象前置

虽然首版不会承诺各平台都达到同等成熟度，但运行时结构必须从一开始避免和单一平台深度耦合。窗口、输入、surface 生命周期等平台差异应收敛到抽象层中。

### 2.6 示例同步交付

每完成一个对外可感知的新功能，都必须在 `examples/` 目录提供对应示例，作为该功能的使用说明、回归样例和验收材料。示例不是文档完成后的补充项，而是每个阶段交付物的一部分。

## 3. v1 范围

v1 明确包含以下能力：

- 应用生命周期管理：应用启动、事件循环、窗口关闭、重绘请求。
- 窗口管理：单窗口优先，多窗口能力预留接口。
- 基础控件树：支持树状 UI 结构、节点组合与统一遍历。
- 响应式属性绑定：将 `ViewModel` 状态映射到控件属性。
- 命令绑定：将用户事件路由到 `ViewModel` 命令入口。
- 布局系统：至少支持 `Stack`、`Row`、`Column`、`Grid`、`Flex` 及基础对齐。
- 渲染管线：基于 `wgpu` 建立稳定的 2D UI 渲染主链路。
- 输入事件：鼠标、键盘、窗口尺寸变化等基础事件接入。
- 状态同步：状态变更触发脏标记、布局刷新与重绘。
- 简单主题能力：提供颜色、字体、间距等基础样式入口。
- 字体系统：支持通过 `include_bytes!()` 加载多个字体资源、为字体命名、设置默认字体，并在文本缺字时自动进行字体回退。

## 4. 非目标

以下内容不纳入 v1 承诺范围：

- 可视化设计器或拖拽式界面编辑工具。
- 完整复杂动画系统与时间轴系统。
- 全量无障碍支持能力。
- Web、移动端或嵌入式端适配。
- 成熟的富文本排版系统。
- 完整资源打包与热更新工作流。

这些能力可以在架构上预留扩展点，但不作为首版交付指标。

## 5. 里程碑规划

### 里程碑一：基础运行时

目标是建立 `Application + Runtime + Renderer` 的基础骨架。

交付物：

- 应用入口与主循环抽象。
- 基于 `winit` 的窗口生命周期管理。
- 基于 `wgpu` 的 surface 初始化、交换链维护与基础清屏渲染。
- 最小示例窗口可以创建、显示并稳定退出。
- 在 `examples/` 中提供基础窗口运行示例。

### 里程碑二：MVVM 绑定

目标是打通 `ViewModel -> View` 和 `View -> Command` 的数据链路。

交付物：

- `ViewModel` 抽象接口。
- 可观察状态与基础 `Binding<T>` 机制。
- 命令抽象与事件回调接入。
- 状态变更驱动 UI 局部刷新或脏标记。
- 在 `examples/` 中提供状态绑定与命令回流示例。

### 里程碑三：基础控件集

目标是提供最小可用控件系统。

交付物：

- `Widget` 或 `Element` 基础抽象。
- 容器类控件与叶子类控件基础类型。
- 至少包括 `Text`、`Button`、`Input`、`Container` 等示例控件。
- `Button` 或其他需要文本内容的控件统一改为接收 `Text` 组件，而不是直接接收字符串。
- `Text` 组件提供字体、颜色和高级文字属性入口，作为统一文本承载组件。
- 统一事件分发与命中流程。
- 在 `examples/` 中为每个新增基础控件提供独立或组合示例。

### 里程碑四：布局与主题

目标是让 UI 具备结构化排版和可调整样式。

交付物：

- `Stack`、`Row`、`Column`、`Grid`、`Flex` 布局能力。
- 基础尺寸、边距、对齐规则。
- 简单主题系统：颜色、字体、间距、边框等样式入口。
- 字体管理能力：支持注册多个命名字体、默认系统字体、手动设置默认字体以及自动字体回退。
- 布局结果与渲染输出协同工作。
- 在 `examples/` 中为每种布局、主题和字体能力提供对应演示示例。

### 里程碑五：示例与文档

目标是整理前序阶段持续积累的示例，并让框架具备对外展示和继续迭代的基础。

交付物：

- 至少一个完整 MVVM 示例应用，并整理前序功能示例的命名与目录结构。
- 项目结构说明与快速上手文档。
- 关键模块设计说明。
- v1 功能边界与后续路线说明。

## 6. 风险与缓解

### 6.1 `wgpu / winit` 抽象复杂度

风险：

- GPU 初始化、surface 生命周期、窗口事件协调存在较高复杂度。
- 不同平台上对窗口重建、surface 丢失的处理可能不一致。

缓解：

- 将运行时与渲染后端分层。
- 明确 surface 创建、重建、销毁的生命周期边界。
- 优先完成单窗口稳定链路，再逐步扩展能力。

### 6.2 MVVM 与 Rust 所有权模型的适配成本

风险：

- 响应式绑定、回调命令与控件树遍历容易触发借用冲突与生命周期复杂度。

缓解：

- 将 `ViewModel` 的状态访问限制在清晰边界中。
- 使用事件队列、脏标记和命令调度减少深层可变借用。
- 对外暴露的 API 以易用性和安全性为先，不强行追求过度泛型化。

### 6.3 跨平台差异

风险：

- 输入行为、窗口行为、DPI、surface 管理在不同桌面平台存在差异。

缓解：

- 在 runtime/platform 层收敛平台差异。
- 首版重点确保架构不锁死平台扩展。
- 通过平台特定测试与示例逐步补齐兼容性。

### 6.4 声明式 API 易用性

风险：

- Rust 中声明式 DSL 如果设计过重，会带来类型推导困难、宏复杂度过高或调用体验不佳。

缓解：

- v1 优先采用清晰直接的 builder/组合式接口。
- 谨慎引入宏，避免为了“看起来像 DSL”牺牲调试体验。
- 对外文档明确推荐用法和组合模式。

### 6.5 字体管理与文本渲染复杂度

风险：

- 多字体注册、默认字体切换、系统字体查找和缺字回退会显著增加文本渲染链路复杂度。
- `Text` 组件升级为统一文本入口后，`Button`、`Input` 等控件的 API 和内部结构都需要同步调整。

缓解：

- 将字体注册、默认字体选择、回退链计算收敛到独立字体管理模块。
- 约定手动加载字体统一使用 `include_bytes!()`，避免首版引入运行时路径管理复杂度。
- 先完成“系统默认字体 + 命名字体注册 + 回退策略”的稳定闭环，再扩展更复杂的排版能力。

## 7. 验收标准

当 v1 达到以下标准时，可视为项目已经形成最小可用闭环：

- 能运行一个基于 `tgui` 的示例窗口应用。
- `ViewModel` 状态变化可以驱动视图刷新。
- 控件命令可触发交互并回流到 `ViewModel`。
- 控件树支持基础组合、增删与重建。
- 布局系统能稳定计算常见容器场景。
- 渲染主循环稳定，不因常规交互频繁失效。
- 文本绘制能够优先使用手动加载字体，并在未显式加载时回退到系统默认字体。
- 支持注册多个命名字体，并允许设置默认字体。
- 当指定字体、默认字体或系统字体缺少目标文字/图标时，能够自动降级到其他已加载字体；若指定字体名不存在于已加载字体中，则回退到系统字体查找。
- `Button` 等文本控件统一经由 `Text` 组件传递文本内容与样式。
- 每个已交付的核心功能在 `examples/` 中都有可运行的对应示例。
- 文档能指导开发者继续扩展控件与功能模块。

## 8. 实施优先级建议

为减少架构返工，建议后续实现遵循如下优先级：

1. 跑通运行时和渲染基础链路。
2. 建立 `ViewModel`、`Binding<T>`、`Command` 三个核心接口。
3. 实现基础控件树和事件分发。
4. 接入布局系统与最小主题能力。
5. 为每一阶段补齐对应 `examples/` 示例，并同步完善测试和对外文档。

## 9. 总结

`tgui` 的首版方向不是做一个一开始就覆盖全部场景的 GUI 大框架，而是做一个架构干净、状态驱动明确、适合 Rust 生态延展的 GUI 基础平台。只要 v1 能稳定完成“事件输入 -> ViewModel 更新 -> 布局刷新 -> 渲染输出”的闭环，这个项目就已经具备继续演进为完整 GUI 框架的基础。
