# tgui 技术设计报告

## 1. 设计目标

`tgui` 是一个面向桌面平台的 Rust GUI 库，基于 `winit` 负责窗口与事件，基于 `wgpu` 负责图形渲染，并以 MVVM 架构组织 UI 状态与交互逻辑。

本设计报告的目标不是锁死最终代码实现，而是定义一套足够清晰、可指导首版落地的系统设计，使开发者可以直接据此搭建项目骨架与核心模块。

核心约束如下：

- 首版聚焦轻量运行时，不追求一次性覆盖全部 GUI 能力。
- 对外 API 以声明式 DSL 方向为主，底层保留可扩展 `Widget` 抽象。
- 架构上预留桌面三平台能力，但首版以统一抽象与可演进性为优先。

## 2. 架构概览

系统由以下八个核心子系统组成：

### 2.1 Application

负责应用级入口与配置，承担启动运行时、创建主窗口、注册根视图和注入根 `ViewModel` 的职责。

主要职责：

- 作为开发者使用 `tgui` 的起点。
- 持有应用级配置，如窗口标题、初始尺寸、主题入口。
- 启动主循环并接入 Runtime。

### 2.2 Runtime

负责协调平台事件、调度更新、驱动布局和渲染，是整个系统的主控层。

主要职责：

- 接收 `winit` 事件。
- 分发输入与窗口生命周期事件。
- 管理重绘请求、脏标记和帧级调度。
- 协调 UI Tree、Binding、Layout、Renderer 的执行顺序。

### 2.3 Renderer

负责将 UI 层生成的绘制描述转化为 GPU 可执行命令，并通过 `wgpu` 提交到目标 surface。

主要职责：

- 初始化 GPU 设备与 surface。
- 管理渲染资源、批处理和提交流程。
- 处理 surface 重建、尺寸变化与渲染失败恢复。
- 协调字体资源、字形缓存和字体回退后的文本绘制提交。

### 2.4 UI Tree

负责维护声明式视图转换后的控件树结构，是布局、事件分发与绘制遍历的基础。

主要职责：

- 表示容器与叶子控件的层级关系。
- 支持节点创建、更新、替换、销毁。
- 为布局计算、命中测试、绘制生成提供统一访问入口。

### 2.5 Binding

负责连接 `ViewModel` 状态与视图属性，是 MVVM 响应式更新的桥梁。

主要职责：

- 订阅 `ViewModel` 中的可观察状态。
- 将状态变化映射到控件属性。
- 触发 UI 节点脏标记和后续刷新链路。

### 2.6 ViewModel

负责封装视图所需状态与命令，不承担底层渲染或平台逻辑。

主要职责：

- 暴露可绑定状态。
- 接收命令调用。
- 协调业务 `Model` 并向视图层输出可消费状态。

### 2.7 Layout

负责计算控件树中每个节点的尺寸与位置，输出可供渲染与命中系统使用的布局结果。

主要职责：

- 实现基础布局规则。
- 根据约束、内容和容器规则确定控件位置与大小。
- 在节点脏化时局部或整体重算布局。

### 2.8 Event

负责统一定义输入事件、命中测试结果和命令触发路径。

主要职责：

- 规范鼠标、键盘、焦点、窗口相关事件类型。
- 将底层平台事件转换为框架内部事件。
- 通过控件树分发到目标节点或命令处理器。

## 3. MVVM 模型定义

### 3.1 Model

`Model` 代表业务数据与领域规则，可来自内存状态、异步任务结果或外部服务。`tgui` 不强绑定具体数据源，只要求 `ViewModel` 能安全消费它们。

### 3.2 View

`View` 负责声明 UI 结构、组件组合与绑定关系，不直接承担复杂业务逻辑。它的核心职责是描述“界面应该长什么样、绑定什么状态、触发什么命令”。

### 3.3 ViewModel

`ViewModel` 负责将 `Model` 转换成适合界面消费的状态，并暴露命令入口。它是 UI 可观察状态的来源，也是用户交互的主要业务入口。

### 3.4 绑定方向

首版采用以下绑定方向：

- 主方向：`ViewModel -> View`
- 交互回流：`View -> Command -> ViewModel`

这意味着：

- 文本、颜色、可见性、启用状态等 UI 属性优先由 `ViewModel` 驱动。
- 点击、输入、选择等用户交互通过命令通道回到 `ViewModel`。
- 不鼓励控件直接读写业务模型，避免视图层承担业务职责。

## 4. 对外 API 方向

`tgui` 的对外 API 以声明式 DSL 为主，但首版不强行锁死具体 Rust 语法形式。设计重点在能力边界，而不是宏或 builder 的最终写法。

计划中的公共接口方向如下：

### 4.1 `App` / `Application`

用于启动运行时和窗口主循环。

设计意图：

- 提供应用入口。
- 持有全局配置。
- 负责根视图与根 `ViewModel` 的组装。

### 4.2 `ViewModel`

以 trait 或等价抽象存在，提供可观察状态与命令入口。

设计意图：

- 定义统一的状态读取与命令处理能力。
- 支持 UI 订阅状态变更。
- 保持业务逻辑与渲染细节隔离。

### 4.3 `Widget` / `Element`

作为声明式 UI 节点抽象，支持容器节点和叶子节点的统一表示。

设计意图：

- 支持树状组合。
- 支持布局、命中测试和绘制遍历。
- 允许后续扩展自定义控件。

### 4.4 `Text`

作为统一文本组件存在，用于承载所有需要显示文字的场景，包括独立文本、按钮标题、输入框内容展示等。

设计意图：

- 统一文本样式和文本绘制入口。
- 支持指定字体、颜色和其他高级文字 API。
- 作为 `Button`、`Input` 等控件内部文本内容的标准输入类型。

### 4.5 `Binding<T>`

作为属性绑定抽象，表示某个 UI 属性依赖于 `ViewModel` 状态。

设计意图：

- 统一表达状态订阅与属性投影。
- 状态变更时自动驱动节点脏化。
- 为文本、布尔开关、颜色、尺寸等属性提供一致机制。

### 4.6 `Command`

作为 MVVM 交互入口，连接用户动作与 `ViewModel` 命令执行。

设计意图：

- 为点击、选择、输入确认等事件提供标准回流接口。
- 将事件分发与业务执行解耦。
- 支持未来扩展同步/异步命令模型。

### 4.7 `FontManager`

作为字体管理抽象，负责字体注册、命名、默认字体选择和字体回退链决策。

设计意图：

- 支持通过 `include_bytes!()` 注册多个字体资源。
- 要求每个手动加载字体拥有稳定名称。
- 默认使用系统字体，并允许应用显式设置默认字体。
- 当指定字体或默认字体缺字时，自动尝试其他已加载字体，再回退到系统字体。

### 4.8 `RendererBackend`

作为渲染层内部或半公开抽象，用于隔离 UI 框架与 `wgpu` 后端细节。

设计意图：

- 限制渲染实现细节的扩散。
- 为未来渲染优化与后端调整预留空间。
- 让 UI 层只依赖绘制命令，不依赖 GPU 资源细节。

## 5. 数据流与刷新机制

`tgui` 首版采用统一帧级刷新机制，以脏标记驱动布局和渲染更新。标准链路如下：

`input -> command/state mutation -> diff/dirty mark -> layout -> render`

具体过程如下：

1. 平台事件进入 Runtime。
2. Runtime 将底层事件转换为框架内部 `Event`。
3. 事件通过命中测试和分发机制到达目标 `Widget` 或命令入口。
4. 命令调用 `ViewModel`，引发状态变化。
5. Binding 系统订阅到状态变化并更新对应属性映射。
6. 相关 UI 节点被标记为 dirty。
7. Runtime 在下一次刷新阶段执行布局重算。
8. 文本节点在生成绘制描述前向字体管理层查询目标字体与回退链。
9. UI Tree 生成中间绘制描述。
10. Renderer 将绘制描述转换为 GPU 命令并提交到 surface。

### 5.1 为什么采用脏标记

脏标记机制可以避免每次事件都全量重建所有状态，有助于平衡易实现性与性能：

- 对小规模界面足够直接。
- 对后续局部刷新优化兼容性较好。
- 适合首版先建立清晰的更新顺序。

### 5.2 Diff 策略

首版不强求复杂虚拟 DOM 式 diff。更现实的策略是：

- 先保证节点更新与替换机制清晰。
- 在必要时对属性更新、布局失效和重绘范围进行细粒度优化。
- 保留未来引入更高阶 diff 策略的空间。

## 6. 控件系统设计

控件系统是 MVVM 视图层的承载结构，设计重点是统一树结构、职责清晰、易于扩展。

### 6.1 基础模型

控件分为两类：

- 容器控件：持有子节点并参与布局规则控制。
- 叶子控件：通常直接输出内容或交互能力，如文本、按钮、输入框、图标等。

其中 `Text` 是首版的核心叶子组件之一，负责统一承载文本内容与文本样式。`Button` 或其他需要传入文字的控件，不再直接接收字符串，而是统一接收 `Text` 组件或等价文本节点。

### 6.2 核心能力

每个 `Widget` 至少需要具备以下能力：

- 描述自身属性。
- 参与布局计算。
- 接收事件或声明不处理事件。
- 生成绘制描述。
- 提供命中测试边界。

对文本相关控件，还需要支持：

- 访问 `Text` 组件提供的文本属性。
- 将字体请求传递给字体管理与文本渲染层。
- 在字体回退后保持布局与绘制结果一致。

### 6.3 树形组合

UI Tree 是控件系统的统一承载结构，应支持：

- 根节点挂载与替换。
- 子节点插入、删除、重排。
- 节点属性更新与局部失效。
- 统一遍历接口，供布局、事件、渲染复用。

### 6.4 事件分发

事件分发需结合命中测试结果进行：

- 先定位事件目标节点。
- 再根据控件能力决定是否消费事件。
- 必要时支持向父节点冒泡或向运行时回落。

首版可以先实现清晰直接的分发模型，不必过早引入复杂捕获阶段。

## 7. 布局系统设计

### 7.1 v1 布局目标

v1 布局系统优先追求稳定和易理解，避免引入复杂约束求解器。首版至少支持：

- `Stack`
- `Row`
- `Column`
- `Grid`
- `Flex`
- `basic alignment`

### 7.2 布局职责

布局系统需要根据以下信息输出结果：

- 父容器约束。
- 节点期望尺寸。
- 间距、边距、填充与对齐规则。
- 子节点顺序与网格位置。
- `Flex` 主轴/交叉轴、换行、伸缩因子等规则。

### 7.3 设计取舍

首版不引入复杂自动布局求解，而是采用规则明确的容器布局模型，原因如下：

- 更贴合最小可用目标。
- 更容易调试和验证。
- 更适合和脏标记、局部刷新策略结合。

其中 `Flex` 作为首版重点容器布局之一，用于覆盖比 `Row` / `Column` 更通用的线性排布场景。设计上可将 `Row`、`Column` 视为更易用的语义化封装，而 `Flex` 提供方向、换行、对齐、伸缩等更完整的控制能力。

## 8. 渲染系统设计

### 8.1 基本思路

渲染系统基于 `wgpu` 构建，UI 层不直接操作 GPU 资源，而是输出中间绘制指令（draw commands / paint primitives），由 Renderer 统一负责批处理和提交。

### 8.2 分层职责

- UI 层：生成可绘制的中间表示。
- Renderer：管理 GPU 设备、管线、缓冲、纹理与提交命令。
- Platform Runtime：提供窗口 surface 与尺寸变化事件。

### 8.3 首版渲染范围

首版优先支持：

- 纯色背景与基本几何块渲染。
- 文本渲染最小实现，能够使用系统字体或手动注册字体进行绘制。
- 基础裁剪与层级绘制顺序。
- 窗口 resize 后的 surface 重建。
- 缺字时按字体回退链进行自动降级绘制。

### 8.4 设计重点

渲染系统要尽量保持以下边界：

- Renderer 不感知业务状态。
- UI Tree 不直接持有 GPU 资源生命周期。
- 绘制命令格式尽量稳定，方便后续做缓存和批处理优化。

### 8.5 字体加载与回退设计

首版字体能力采用“系统字体为默认基础，手动加载字体为增强来源”的策略。

#### 手动加载字体

- 手动加载字体时，字体数据来源统一使用 `include_bytes!()`。
- 每个加载的字体都必须指定唯一名称。
- 字体注册后进入统一字体管理器，供 `Text` 和文本控件查询。

#### 默认字体策略

- 未手动设置默认字体时，文本绘制默认使用系统字体。
- 应用可以将某个已加载字体名称设置为默认字体。
- 当设置默认字体时，如果该名称不在已加载字体中，则回退到系统字体查找，而不是直接报错中断。

#### 指定字体策略

- `Text` 组件可以显式指定字体名称。
- 如果指定的字体名称存在于已加载字体中，则优先使用该字体。
- 如果指定的字体名称不存在于已加载字体中，则退回到系统字体中查找可用字体。

#### 字体回退策略

当系统字体或手动设置的默认字体不存在某个文字或图标时，应自动降级到其他已加载字体中查找可用字形。推荐回退顺序如下：

1. `Text` 显式指定字体。
2. 手动设置的默认字体。
3. 系统默认字体。
4. 其他已加载字体中的可用字体。

实现时可以根据平台字体能力调整查询顺序，但必须满足以下行为要求：

- 缺字不能直接中断整段文本绘制。
- 回退过程应对上层 `Text` API 透明。
- 同一段文本允许按字形粒度落到不同字体，但首版可优先实现按片段分段回退。

### 8.6 文本组件能力

`Text` 组件不再只是简单字符串承载器，而应成为统一文字渲染配置入口。首版至少应支持：

- 文本内容。
- 字体名称。
- 颜色。
- 字号、字重、字距等可扩展高级文字 API 入口。

其他文本控件的文字参数应尽量统一建模为 `Text`，以减少按钮、标签、输入框等组件各自维护一套文本样式参数的重复设计。

## 9. 平台抽象设计

### 9.1 平台来源

窗口、输入与事件循环来自 `winit`，这是首版平台交互的主要来源。

### 9.2 抽象边界

平台差异应主要收敛在 `runtime/platform` 层，而不是散落在控件、绑定或布局系统内。

应重点隔离的差异包括：

- 窗口生命周期。
- DPI 与逻辑尺寸转换。
- surface 创建与重建时机。
- 输入行为差异。

### 9.3 平台目标

项目目标是桌面三平台，但首版设计要求是“结构预留、边界清晰、实现逐步完善”。也就是说：

- 文档层面明确 Windows、macOS、Linux 是目标平台。
- 实现层面优先保证抽象可扩展。
- 不在 v1 文档中承诺所有平台完全同等成熟。

## 10. 错误处理与生命周期

首版需要明确以下关键流程：

### 10.1 初始化失败

包括窗口创建失败、GPU 设备初始化失败、surface 创建失败等。这些错误应在应用启动阶段被清晰上报，而不是静默吞掉。

### 10.2 资源创建失败

包括渲染管线、缓冲区、纹理或字体资源创建失败。应尽量统一成框架级错误类型，便于诊断。

### 10.3 窗口关闭

当收到关闭请求时，Runtime 需要完成退出流程，确保事件循环、安全释放渲染资源与必要清理。

### 10.4 Surface 重建

窗口尺寸变化、surface 丢失或平台事件导致的重建，是 `wgpu` 场景中的常见流程。首版需要把这部分生命周期明确建模，而不是散落在渲染代码中临时处理。

## 11. 扩展性设计

虽然这些能力不纳入 v1 承诺，但设计上应保留清晰扩展点：

- 主题系统：扩展颜色、字体、边框、阴影、状态样式。
- 动画系统：允许未来为属性变化引入时间维度。
- 资源系统：支持图片、字体、图标等外部资源管理。
- 自定义控件：允许第三方扩展 `Widget` 能力。
- 异步任务：支持将后台任务结果安全回流到 `ViewModel`。

这些扩展点的目标不是提前做完，而是避免首版架构把未来能力堵死。

## 12. 建议的模块拆分

为了便于后续代码实现，建议项目骨架优先拆为以下模块：

- `app`：应用入口与配置。
- `runtime`：事件循环、调度、平台桥接。
- `renderer`：`wgpu` 渲染后端。
- `text` 或 `font`：字体注册、默认字体、回退策略与文本样式抽象。
- `widget`：控件抽象与基础控件。
- `tree`：UI 树结构与节点管理。
- `binding`：状态绑定与命令系统。
- `view_model`：`ViewModel` 抽象与辅助类型。
- `layout`：布局计算。
- `event`：事件定义与分发。
- `theme`：基础主题能力。

该拆分是建议性的，最终实现可以合并部分模块，但不建议把运行时、渲染、绑定与控件系统混在同一层中。

## 13. 示例策略

`examples/` 目录应被视为框架开发流程的一部分，而不是发布前补充材料。每完成一个新功能，都需要补一个对应示例，用来承担三类职责：

- 用法展示：说明开发者应该如何使用该能力。
- 回归验证：作为功能完成后的最小可运行验证样例。
- 设计校验：通过真实调用方式反向检查 API 是否易用。

建议按功能组织示例，例如：

- 基础运行时示例。
- `Binding` / `Command` 示例。
- 基础控件示例。
- `Input` 输入与状态同步示例。
- 多字体加载、默认字体切换与自动回退示例。
- `Text` 高级文字 API 与文本控件组合示例。
- `Stack`、`Row`、`Column`、`Grid`、`Flex` 各自的布局示例。
- 主题能力示例。
- 完整 MVVM 综合示例。

如果某个功能还无法写出清晰、可运行的示例，通常意味着该功能的对外接口、默认行为或依赖关系仍不够稳定，需要在实现时继续收敛设计。

## 14. 示例接口草案

以下示意仅用于表达设计方向，不作为最终 Rust 语法承诺：

```rust
let app = Application::new()
    .title("Counter")
    .window_size((800, 600))
    .font("ui", include_bytes!("../assets/fonts/YourUIFont.ttf"))
    .font("icons", include_bytes!("../assets/fonts/YourIcons.ttf"))
    .default_font("ui")
    .view_model(CounterViewModel::default())
    .root_view(|vm| {
        Column::new()
            .child(
                Text::new(vm.bind(|s| s.count_text()))
                    .font("ui")
                    .color(Color::rgb(0.1, 0.1, 0.1))
            )
            .child(
                Button::new(
                    Text::new("Increment")
                        .font("ui")
                )
                .on_click(vm.command("increment"))
            )
    });

app.run()?;
```

这个示例表达的重点是：

- 应用入口统一。
- 字体通过 `include_bytes!()` 注册并按名称管理。
- 根视图由声明式组合构建。
- 文本通过 `Binding<T>` 从 `ViewModel` 获取，并由 `Text` 组件统一承载样式。
- `Button` 等文本控件统一接收 `Text` 组件。
- 按钮点击通过 `Command` 回流到 `ViewModel`。

## 15. 总结

`tgui` 的技术路线应围绕一个清晰而稳定的核心闭环展开：

- Runtime 接收事件。
- ViewModel 管理状态与命令。
- Binding 将状态映射到 View。
- Layout 计算结构。
- Renderer 完成输出。

只要这个闭环建立稳固，`tgui` 就能够以较低风险逐步扩展成完整的 Rust GUI 框架。首版最重要的不是做多，而是把模块边界、数据流和扩展方向设计正确。
